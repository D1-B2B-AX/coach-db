# 실습코치 구인 프로세스 — 병목 분석 및 문제 정의

> 운영매니저 3인 인터뷰 기반 | 문제 정의 및 솔루션 설계를 위한 컨텍스트 문서

---

## 1. 배경 및 현황

운영매니저는 교육 과정마다 실습코치를 섭외해야 한다.

**현재 프로세스 순서:**
1. 기존 보유 코치 명단(노션)과 부가 정보(구글 스프레드시트) 검토
2. 적임자가 있으면 개별 연락 및 일정 조율
3. 적임자가 없으면 구직 사이트에 공고 게시 후 신규 코치 모집

**문제의 핵심:** 코치 활동 이력이 노션에 꾸준히 업데이트되지 않고, 관련 데이터가 여러 툴에 분산되어 있어 구인에 드는 시간과 인력 낭비가 크다.

### 현재 데이터가 저장된 곳

| 툴 | 저장 데이터 |
|---|---|
| 노션 | 코치 명단, 포트폴리오, 커리큘럼 |
| 구글 스프레드시트 | 일부 코치 정보, 계약 및 강사비 내역 |
| 구글 드라이브 | 포트폴리오 파일 |
| 구글 설문 | 만족도 조사 결과 (코치 DB와 미연결) |

---

## 2. 핵심 병목 3가지

### 🔴 병목 1 — 데이터 파편화: "어디에 있는지 모른다"

코치 정보가 노션·구글시트·드라이브에 분산되어 있고, 팀 간(1팀↔2팀) 공유도 단절되어 있다. 한 팀이 보유한 우수 코치를 다른 팀이 알지 못하는 상황이다.

- 코치가 법인인지 개인인지, 신규인지 기존인지도 한눈에 파악되지 않는다
- 강사 DB와 계약 설문지가 분리되어 있어 신규 등록 시 여러 곳을 직접 확인해야 한다
- 만족도 조사 결과가 구글 설문에 쌓이지만 코치 DB와 매핑이 안 되어 활용되지 못하고 있다

> *"데이터 최종본이 어디 있는지 모르겠다. 어디 한군데에 다 있었으면 좋겠다."* — 사용자1

---

### 🟡 병목 2 — 일정·가용 현황 불투명: "직접 물어봐야 안다"

코치가 현재 활동 중인지, 가용한지를 확인하려면 담당 팀·코치 본인·담당자에게 직접 연락해야 한다. 1팀과 2팀의 일정이 산발적으로 관리되어 통합 조회가 불가능하다.

- 기존 숙련 코치들이 특정 전용 과정에 고정 배치된 상태라 신규 과정에 배정할 여유 인력 파악이 어렵다
- 코치 가능 툴·가능 일정 정보가 노션에 업데이트되지 않아 수작업으로 확인해야 한다

> *"활동 중인지 아닌지 일일이 수작업 업데이트가 불편하다. 코치 1명 확인하는 데 여러 명에게 물어봐야 한다."* — 사용자2·3

---

### ⚪ 병목 3 — 신규 리크루팅 허들: "공고 올려도 지원이 없다"

체계적인 DB가 없어 신규 코치 발굴·배치에 드는 리소스가 과도하다. 공고를 올려도 지원이 저조하고, 다수에게 섭외 메일을 일일이 보내야 한다.

- 포트폴리오를 노션·드라이브에서 각각 열어봐야 해서 후보 검토 자체가 비효율적이다
- 키워드 요약 없이 포트폴리오 전문을 읽어야 한다

> **사용자3 주석:** 시스템이 구축되면 파편화와 일정 불투명 문제는 90% 이상 해결 가능하지만, 신규 리크루팅은 결이 다른 문제다. DB가 갖춰지더라도 지원자 자체가 부족한 구조적 문제가 남는다.
>
> *"사람이 없어서 못 구하는 게 아니라, 있는 사람을 몰라서 못 구하는 상황이 먼저 해결되어야 한다."* — 사용자3

## 3. 현재 구인 플로우 (As-Is)

```
과정 오픈
  └─> 노션 명단 확인 + 구글시트 확인   ← 각 툴을 따로 열어서 수동 대조
        └─> 포트폴리오 열람              ← 노션 + 드라이브 각각 열람
              └─> 활동 여부 수동 확인    ← 담당자·코치에 직접 연락
                    └─> 섭외 메일 개별 발송
                          ├─> (적임자 있음) 일정 조율 → 계약
                          └─> (적임자 없음) 구직사이트 공고 → 지원자 검토 → 계약
```

## 4. 사용자 요구사항 정리

### 필수 (Must Have)

- 코치 정보(이름·연락처·가능 툴·일정·계약이력)를 **한 곳에서** 볼 수 있어야 한다
- 현재 **가용 여부를 즉시** 확인 가능해야 한다 (수작업 업데이트 없이)
- 코치 과거 이력·만족도·강점 분야를 **프로필 한 페이지**에서 확인

### 중요 (Should Have)

- 팀 구분 없이 **전체 코치 풀 공유** (1팀↔2팀 데이터 통합)
- 섭외 시 다수에게 **대량 메일 발송** 기능 (현재 개별 발송 중)
- 코치별 **키워드 태그** (가능 툴, 강점 분야 요약) — 포트폴리오 전문 열람 없이 판단 가능하도록

### 있으면 좋음 (Nice to Have)

- 커리큘럼 기반 **코치 자동 추천** (노션 커리큘럼 → 매칭 코치 제안)
- 계약 완료 시 **슬랙 알림** 자동 발송
- 코치 **자가 일정 업데이트** 기능 (코치가 직접 가용 여부 등록)

### 장기 검토

- 신규 코치 **지원서 수령 자동화** — 같은 페이지 내에서 지원 → 등록 일원화

## 5. 설계 시 고려사항

### 포함 (In Scope)

- 코치 통합 DB 구축
- 일정 및 가용 현황 조회
- 프로필 + 이력 + 만족도 통합
- 팀 간 데이터 공유 구조
- 섭외 메일 대량 발송 기능
- 주관적 메모 입력 기능 (코치에 대한 정성적 평가 기록)

### 주의 / 별도 논의 필요

- 신규 리크루팅 자체는 DB 구축과 결이 다른 문제 — 별도 접근 필요
- 사용자들이 노션보다 구글시트를 선호하는 경향 → 플랫폼 선택 시 반영 필요
- 코치 자가 업데이트 설계 시 접근 권한 관리 방안 필요
- 만족도 설문과 코치 DB 간 매핑 구조 설계 필요

## 6. 다음 단계 제안

1. **통합 DB 구조 설계** — 코치 프로필에 포함될 핵심 필드 정의 (기본정보, 계약정보, 이력, 만족도, 일정, 키워드 태그)
2. **플랫폼 결정** — 노션 vs 구글시트 vs 별도 툴(에어테이블 등). 사용자 선호 및 운영 편의성 고려
3. **기존 데이터 마이그레이션 계획** — 노션·구글시트·구글 설문 데이터를 어떻게 통합할지
4. **우선순위 결정** — 파편화 해소 + 일정 가시화를 1단계로, 신규 리크루팅 자동화는 2단계로
