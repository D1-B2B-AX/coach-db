# Clarify → Build 반복 작업 방식

> 이 문서는 프로젝트의 문서 구조와 작업 흐름을 설명한다.

## 핵심 원칙

**git이 추적하는 것:** 코드가 어떻게 바뀌었는지 (what)
**문서가 추적하는 것:** 왜 그렇게 바뀌었는지 (why)

코드 변경은 git이 기억한다. 문서는 git이 기억하지 못하는 것만 기록한다.

---

## 문서 구조

```
coach/
├── CLAUDE.md                              ← 세션 진입점
└── docs/plans/
    ├── how-we-work.md                     ← 이 문서
    ├── decisions.md                       ← 의사결정 로그
    ├── YYYY-MM-DD-*-design.md             ← 설계 문서
    └── YYYY-MM-DD-*-implementation.md     ← 구현 계획
```

### 1. CLAUDE.md — 세션 진입점

- 프로젝트 루트에 위치. Claude Code가 매 세션 시작 시 자동으로 읽는다.
- 현재 상태, 주요 문서 위치, 핵심 컨벤션을 담는다.
- **역할:** 새 세션을 열었을 때 "지금 어디까지 왔고, 뭘 봐야 하는지"를 즉시 파악하게 해준다.

### 2. decisions.md — 의사결정 로그

- 모든 Clarify 라운드에서 내린 결정을 기록한다.
- 각 항목에는 **결정 내용**, **이유**, **영향 범위**를 적는다.
- **역할:** "왜 이렇게 만들었지?"라는 질문에 답한다. 2주 뒤의 나, 또는 합류한 팀원이 맥락을 빠르게 파악할 수 있다.
- **작성 기준:** 코드나 git diff만으로 이유를 알 수 없는 결정만 기록한다.

### 3. design doc — 설계 문서

- 현재 합의된 설계의 최신본이다.
- Build 중 스펙이 바뀌면 **이 문서를 직접 수정**한다. 원본을 보존하려 하지 않는다 — git이 원본을 기억한다.
- **역할:** "지금 뭘 만들고 있는지"의 정답이다.

### 4. implementation plan — 구현 계획

- 설계를 코드로 옮기기 위한 Task 목록이다.
- 각 Task에 파일 경로, 코드, 커밋 단위가 포함되어 있다.
- 상단의 체크박스로 진행률을 추적한다.
- **역할:** "다음에 뭘 해야 하는지"의 정답이다.

## 작업 흐름

```
┌──────────┐     ┌──────────┐     ┌──────────┐
│ Clarify  │────▶│  Build   │────▶│  Review  │──┐
│          │     │          │     │          │  │
│ 스펙 논의  │     │ Task 구현  │     │ 동작 확인  │  │
│ 결정 기록  │     │ 체크박스 ✅ │     │ 피드백    │  │
└──────────┘     └──────────┘     └──────────┘  │
     ▲                                          │
     └──────────────────────────────────────────┘
                  스펙 변경 시 다시 Clarify
```

### Clarify 단계에서 하는 일

1. 스펙에 대해 질문하고 결정을 내린다
2. `decisions.md`에 결정 기록
3. 설계 문서 업데이트 (변경된 부분만)
4. 필요 시 구현 계획의 Task 수정/추가/삭제

### Build 단계에서 하는 일

1. 구현 계획의 Task를 순서대로 실행
2. Task 완료 시 체크박스 체크 (`- [x]`)
3. 커밋
4. Build 중 스펙 의문이 생기면 → Clarify로 돌아간다

### Review 단계에서 하는 일

1. 구현된 기능 동작 확인
2. 피드백이 있으면 → Clarify로 돌아간다
3. OK면 → 다음 Task로 진행

## 문서 갱신 타이밍 요약

| 이벤트 | 갱신 대상 |
|--------|-----------|
| 스펙 결정/변경 | decisions.md + design doc |
| Task 완료 | implementation plan 체크박스 |
| 단계 전환 (Clarify↔Build) | CLAUDE.md 현재 상태 |
| 새 세션 시작 | 없음 (CLAUDE.md가 자동 로드) |

## 하지 않는 것

- 설계 문서의 이전 버전을 별도 파일로 보관하지 않는다 (git log로 충분)
- 코드에서 읽을 수 있는 내용을 문서에 중복 기록하지 않는다
- 디버깅 과정이나 임시 메모를 문서에 남기지 않는다
